## MCP와 Skills의 관계

### 레이어 구분

MCP와 Skills는 서로 다른 계층에서 동작합니다.

MCP는 **도구 제공 계층**입니다. LLM이 외부 시스템과 통신할 수 있는 인터페이스를 정의합니다. Slack 메시지 조회, DB 쿼리, 파일 시스템 접근 같은 개별 기능을 tool로 노출하는 역할입니다.

Skills는 **작업 지시 계층**입니다. LLM이 도구를 포함한 모든 능력을 어떤 순서로, 어떤 규칙을 지키며 사용해야 하는지를 정의합니다.

정리하면 MCP는 "무엇을 할 수 있는가", Skills는 "어떻게 해야 하는가"입니다.

### 조합 패턴

스킬 내부에서 MCP 도구를 참조하는 형태로 조합됩니다.

예시로 "고객 문의 분석 리포트" 스킬이 있다고 하면:

```markdown
## Instructions

1. MCP(Slack)로 최근 7일간 #support 채널 메시지를 조회
2. 문의를 버그/기능요청/일반질문으로 분류
3. 분류 결과를 MCP(Google Sheets)에 기록
4. 요약 리포트를 MCP(Gmail)로 발송
```

스킬이 워크플로우를 정의하고, 각 단계에서 MCP 도구를 호출하는 구조입니다. 스킬이 MCP 도구들의 오케스트레이터 역할을 합니다.

### 각각만 있을 때의 한계

**MCP만 있고 Skills가 없는 경우** — 도구는 쓸 수 있지만 활용 품질이 일정하지 않습니다. 같은 요청에도 매번 다른 순서, 다른 형식으로 결과가 나올 수 있습니다. LLM이 매번 즉흥적으로 판단하기 때문입니다.

**Skills만 있고 MCP가 없는 경우** — LLM 내장 능력(텍스트 생성, 코드 작성 등)만 활용 가능합니다. 외부 데이터 조회나 시스템 연동이 필요한 작업은 수행할 수 없습니다.

### 설계 시 고려사항

스킬에서 MCP 도구를 참조할 때는 **도구 이름을 직접 하드코딩하기보다 역할 기반으로 기술**하는 것이 유연합니다. 예를 들어 "Slack의 get_messages를 호출"보다 "메시징 도구로 최근 메시지를 조회"로 쓰면, MCP 서버가 바뀌어도 스킬을 수정할 필요가 줄어듭니다.

또한 스킬의 description에 어떤 MCP 도구가 필요한지 명시해두면, 해당 도구가 연결되지 않았을 때 LLM이 사전에 안내할 수 있어서 사용자 경험이 좋아집니다.
