## A2A (Agent-to-Agent) 프로토콜 정리

**A2A란?**

A2A(Agent2Agent)는 Google이 2025년 4월에 발표한 오픈 프로토콜로, AI 에이전트들이 서로 통신하고 안전하게 정보를 교환하며 다양한 엔터프라이즈 플랫폼에서 작업을 조율할 수 있도록 설계되었습니다. 2025년 6월에는 Linux Foundation 산하 오픈소스 프로젝트로 편입되어 벤더 중립적인 거버넌스 하에 운영되고 있습니다.

**MCP와의 관계**

A2A는 Anthropic의 MCP(Model Context Protocol)와 경쟁이 아닌 보완 관계입니다. 공식 문서에서도 "MCP는 도구(tools)에, A2A는 에이전트(agents)에 사용하라"고 권장합니다. 쉽게 비유하면, MCP는 에이전트가 데이터베이스나 API 같은 도구에 접근하는 방식을 표준화하고, A2A는 에이전트끼리 대화하고 협업하는 방식을 표준화합니다. 예를 들어 소매점의 재고 에이전트가 MCP로 제품 데이터베이스에 접근하고, 재고 부족을 감지하면 A2A를 통해 외부 공급업체 에이전트와 소통해 주문을 넣는 식입니다.

**핵심 설계 원칙 5가지**

Google은 파트너들과 함께 다음 원칙에 따라 프로토콜을 설계했습니다.

1. **에이전트의 자율성 존중** — 에이전트를 단순 "도구"로 제한하지 않고, 메모리·도구·컨텍스트를 공유하지 않아도 자연스럽게 협업할 수 있도록 함
2. **기존 표준 활용** — HTTP, SSE(Server-Sent Events), JSON-RPC 등 이미 널리 쓰이는 웹 표준 위에 구축
3. **기본적으로 안전** — 엔터프라이즈급 인증·인가를 지원하며 OpenAPI의 인증 체계와 동등한 수준
4. **장기 실행 작업 지원** — 즉시 완료되는 작업부터 수 시간~수 일 걸리는 심층 리서치까지 유연하게 처리
5. **불투명성(Opacity) 보장** — 에이전트가 내부 상태, 메모리, 도구 구현을 노출하지 않고도 협업 가능

**핵심 구성 요소**

A2A 클라이언트(요청하는 에이전트)와 A2A 서버(작업을 수행하는 원격 에이전트) 간의 통신으로 구성됩니다. 주요 구성 요소는 다음과 같습니다.

- **Agent Card** — 에이전트의 이름, 설명, 버전, 서비스 엔드포인트 URL, 지원 모달리티, 인증 요구사항 등을 담은 JSON 파일로, 에이전트 간 상호 발견(discovery)을 가능하게 합니다.
- **Task** — 클라이언트와 원격 에이전트 간 통신의 기본 단위입니다. 생성~완료(또는 실패/취소)까지 라이프사이클을 가집니다.
- **Artifact** — 작업의 출력물입니다.
- **Message & Parts** — 각 메시지는 "파트"로 구성되며, 파트는 생성된 이미지 등 완전한 형태의 콘텐츠 조각입니다. 이를 통해 에이전트들이 적절한 형식과 UI 능력을 협상합니다.

**통신 방식**

JSON-RPC 2.0 over HTTP(S)를 기본 통신 표준으로 사용하며, 동기 요청/응답, SSE 스트리밍, 비동기 푸시 알림을 모두 지원합니다.

**업계 지원 현황**

100개 이상의 주요 기술 기업이 참여하고 있으며, Atlassian, Salesforce, SAP, PayPal, ServiceNow 등 엔터프라이즈 대표 기업들이 포함되어 있습니다. Microsoft도 Azure AI Foundry와 Copilot Studio에 A2A 지원을 예고했고, AWS 역시 프로젝트 기여 의사를 밝혔습니다.

**보안 관련 주의점**

Agent Card에 대한 중앙 레지스트리가 없어 스푸핑 위험이 있고, OAuth 토큰 수명 제한이 강제되지 않으며, 에이전트 간 민감 데이터 공유 시 사용자 동의 메커니즘이 프로토콜 수준에서 요구되지 않는 등 아직 보안 측면에서 보완이 필요한 부분이 있습니다. 프로토콜 자체가 아직 발전 중(WIP)이라 향후 개선이 기대됩니다.

요약하면, A2A는 "에이전트들의 공용어"를 만들려는 시도로, MCP가 에이전트와 도구 사이의 다리라면 A2A는 에이전트와 에이전트 사이의 다리입니다. 멀티 에이전트 시스템이 본격화되면서 점점 중요해질 프로토콜이라고 볼 수 있습니다.
